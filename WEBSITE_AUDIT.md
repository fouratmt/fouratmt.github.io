# fourat.dev Website Audit

Audit date: 2026-07-15

## Executive summary

The site is small, fast to build, responsive, and technically stable. The production homepage also loaded without console errors during the spot check. The main weaknesses are not basic rendering problems; they are discoverability, bilingual consistency, accessibility, content depth, and deployment hygiene.

The first fixes should be:

1. Allow the English and French About and CV pages to be indexed if they are intended to attract clients or recruiters.
2. Make every language switch and French homepage link stay on the equivalent translated page.
3. Rewrite the privacy policy so it matches what the deployed site actually does.
4. Fix the green availability text in dark mode and make the CV/PDF experience accessible.
5. Stop publishing empty taxonomy pages, `.DS_Store`, and a reference to a missing icon.

## Implementation status

The repository-actionable findings in this audit were addressed on 2026-07-15 in the dedicated audit-fix branch. The implementation:

- makes the public About and CV pages indexable while keeping Privacy and 404 responses `noindex, follow`;
- preserves the current page when switching languages and localizes shared navigation, footer, and control labels;
- replaces inaccurate privacy wording, unsafe availability styling, vague alternative text, and inaccessible PDF embeds;
- moves source images through Hugo's responsive WebP pipeline, adds a 1200 × 630 social card, emits valid non-visible JSON-LD structured data, and removes unused taxonomy/search output;
- expands the bilingual About content, keeps the CV routes focused on their owner-managed PDFs, improves the PDF embed and download fallback, and leaves the PDF artifacts owned by their external LaTeX build pipeline;
- adds warning-as-error builds, generated-site/accessibility validation, and headless-browser route and mobile smoke checks for pull requests;
- adds a multi-stage unprivileged Docker image, Docker Compose development and production-like local preview services, and HTTP security headers for local container testing.

Two actions remain outside this repository: the equivalent response headers in `docs/security-headers.md` must be enabled in Cloudflare after this branch is deployed, and PDF tagging/metadata should be addressed in the separate LaTeX source project. The public website remains hosted by GitHub Pages; the local-only Docker preview applies its own headers through NGINX, and this project no longer modifies the externally generated PDF artifacts.

At the site owner's request, the header language/theme controls and footer links retain PaperMod's original compact spacing. The audit's 44 × 44px touch-target recommendation therefore remains intentionally unimplemented.

## Scope and method

This audit covered:

- Hugo configuration, bilingual content, custom layouts, static assets, and the GitHub Pages workflow.
- A production-style build with Hugo Extended 0.155.2 using `--gc --minify`, i18n warnings, path warnings, and template metrics.
- Rendered English and French home, About, CV, and Privacy pages at 1440 × 900 and 375 × 812.
- Generated metadata, internal asset references, PDF metadata, basic accessibility checks, and a production homepage spot check at `https://fourat.dev/`.

This was not a full legal review, penetration test, assistive-technology user test, or field-performance study. Production HTTP response headers and Core Web Vitals should be measured separately after the changes are deployed.

## Findings

### P1 — High priority

#### 1. Valuable portfolio pages explicitly tell search engines not to index them

Evidence:

- `content/en/about.md:9`, `content/fr/about.md:8`, `content/en/cv.md:8`, and `content/fr/cv.md:8` set `robotsNoIndex: true`.
- The generated pages contain `meta name="robots" content="noindex, nofollow"`.
- The homepage is indexable, but it contains very little crawlable detail about services, experience, or technical expertise.

Impact: searches for Fourat's experience, freelance services, or CV cannot reliably surface the pages that contain the relevant information. `nofollow` also asks crawlers not to follow the useful links on those pages.

Improvement:

- Remove `robotsNoIndex` from both About pages and, if public discoverability is desired, both CV pages.
- Keep the Privacy page `noindex` if that is intentional, but prefer `noindex, follow` over `noindex, nofollow`.
- Confirm the final behavior in Google Search Console after deployment.

#### 2. The bilingual navigation can send visitors to the wrong language or wrong page

Evidence:

- `layouts/partials/header.html:101` builds the header switch from `site.Home.AllTranslations`; on `/about/` and `/cv/`, the switch therefore leads to the other-language homepage instead of the translated current page.
- `config.yml:57` links the French “et bien plus encore” text to `/about`, which is the English page.
- The content-level translation link generated by PaperMod is correct, so the page currently exposes two language-switch behaviors that disagree.
- The global label in `config.yml:144` renders “Home” in the French header, and `layouts/partials/footer.html:15` renders “Privacy Policy” in French pages.

Impact: French visitors can unexpectedly leave the French experience, while users switching language lose their current context.

Improvement:

- Use the current page's `.AllTranslations` in the header, with a homepage fallback only when no translated page exists.
- Use language-aware URLs such as `relLangURL`, `relref`, or translated menu entries instead of hard-coded `/about` and `/fr/...` paths.
- Move “Home”, “Privacy Policy”, theme labels, and similar UI text into `i18n/en.yaml` and `i18n/fr.yaml`, or define language-specific parameters.
- Add an automated assertion that `/about/` switches to `/fr/about/` and `/fr/cv/` switches to `/cv/`.

#### 3. The privacy policy does not match the observed implementation

Evidence:

- `content/en/privacy.md:15` and `content/fr/privacy.md:15` say that Cloudflare, Google Analytics, and Hugo collect data.
- Neither the local production build nor the deployed homepage included a Google Analytics or Cloudflare analytics script during this audit.
- Hugo is a static-site generator and does not collect visitor data by itself.
- Theme preference is stored in `localStorage` by `layouts/partials/footer.html`, not in a cookie as the longer commented policy text states.
- The policy does not identify the analytics configuration, data categories, legal basis, retention, opt-out mechanism, or contact/controller details.

Impact: an inaccurate notice weakens visitor trust and can create compliance risk. The current wording may both overstate and omit data processing.

Improvement:

- First decide which production services actually process requests or analytics, including any Cloudflare proxy/analytics settings that are not visible in the repository.
- Describe only those services and distinguish essential request logs from optional analytics.
- Remove Hugo from the list of data collectors and describe the `localStorage` theme preference accurately.
- If Google Analytics is enabled later, document consent/opt-out behavior and ensure the CSP permits only the required endpoints.
- Have the final text reviewed for the jurisdictions and business context that apply.

#### 4. Availability text fails normal-text contrast in dark mode

Evidence:

- `config.yml:24` and `config.yml:57` hard-code `color: green` inside HTML subtitles.
- In dark mode, the computed colors were `rgb(0, 128, 0)` on `rgb(29, 30, 32)`, approximately **3.24:1**.
- The text is 18px bold, which is below the WCAG large-text threshold of about 18.67px bold, so the expected AA ratio is 4.5:1.

Impact: the most important status/CTA text is harder to read for low-vision visitors and does not meet WCAG AA for normal text.

Improvement:

- Remove inline color styles and define a semantic availability class in a custom stylesheet.
- Give that class separate light/dark theme colors that reach at least 4.5:1.
- Keep the underline or another non-color cue for the link.

### P2 — Medium priority

#### 5. The CV embed and the PDFs are not accessible enough

Evidence:

- `layouts/shortcodes/pdfReader.html:1` emits an `<embed>` with no accessible title or fallback content.
- Both PDF files report `Tagged: no`, have no document title/author metadata, and are not optimized for fast web view.
- On mobile, the embed creates a 1000px-tall viewer inside a narrow page. It does not overflow horizontally, but native PDF viewers vary significantly by browser.

Impact: screen-reader users may not understand the embedded object, untagged PDFs have poor navigation/reading order, and some mobile visitors get a cumbersome viewer.

Improvement:

- Add a localized `title` and visible download/open link around an `<object>` or `<iframe>` fallback.
- Consider hiding the inline viewer at small breakpoints and making “Open/download PDF” the primary mobile action.
- Export tagged PDFs with a document title, language, author, headings, logical reading order, and accessible links.

#### 6. Several labels, alternative texts, and touch targets need improvement

Evidence:

- `layouts/partials/header.html:77` gives the theme button the accessible name “(Alt + T)” rather than its purpose.
- The language links measured only 15–19px wide, and footer links measured 15px high on the 375px viewport; their interactive boxes are below the commonly recommended 44 × 44px target.
- `config.yml:28` and `config.yml:61` produce the vague profile alt text “fourat”.
- `content/en/about.md:13` and `content/fr/about.md:12` display `logo.png` while describing it as “Picture of me”.
- Links such as “here” and “ici” in the About pages are ambiguous when read out of context.

Impact: controls are harder to understand or tap, especially for screen-reader, motor-impaired, and mobile users.

Improvement:

- Add a localized `aria-label`, such as “Switch to dark theme”, and update it when the theme changes.
- Add padding/minimum dimensions and visible focus styles to compact header/footer controls.
- Use “Portrait of Fourat Mastouri” for the portrait, or an empty alt for decorative images; describe the logo as a logo.
- Replace generic links with “View my LinkedIn profile” and “Read my résumé/CV”.

#### 7. Images bypass Hugo's processing pipeline

Evidence:

- `layouts/partials/header.html:49` and `layouts/partials/index_profile.html:7` call `resources.Get`, which reads from `assets/`.
- The referenced images are in `static/`, while `assets/` is empty, so both templates fall back to the full original files.
- `logo.png` is 274KB at 638 × 632 but is displayed at 35px in the header. `me.jpg` is 108KB at 798 × 767 but is displayed at 150 × 150 on the homepage.
- The About page also loads the 274KB logo at 50% width without explicit intrinsic dimensions.

Impact: visitors download far more image data than required, particularly on mobile, and the About image can contribute to layout shift.

Improvement:

- Move processable source images into `assets/` and let Hugo create correctly sized WebP/AVIF variants.
- Keep only truly static passthrough files in `static/`.
- Add explicit width and height, `srcset`, and `sizes` where an image changes size by viewport.
- Compress the logo or use a compact SVG if the artwork is suitable for vector output.

#### 8. Generated output contains dead pages and an unresolved asset

Evidence:

- Hugo publishes empty, indexable `/tags/`, `/categories/`, `/fr/tags/`, and `/fr/categories/` pages plus their RSS files, even though the site has no taxonomy content.
- The 404 pages render `index, follow` and appear in the normal multilingual metadata flow.
- All 14 generated HTML pages reference `safari-pinned-tab.svg`, but the file does not exist.
- `static/.DS_Store` is copied to the deployment root as `/.DS_Store`.
- Home JSON search indexes are generated even though no search page or navigation entry is present.

Impact: search engines discover thin/dead URLs, users or crawlers request a missing icon, and deployment artifacts contain unnecessary files.

Improvement:

- Disable unused taxonomies in Hugo (`disableKinds` or empty taxonomy configuration) and remove JSON output unless search will be exposed.
- Ensure 404 output is not indexed where the hosting layer permits it.
- Add the missing mask icon or remove the reference.
- Delete `static/.DS_Store` and ignore `.DS_Store` repository-wide.

#### 9. Search and social metadata are generic or semantically inaccurate

Evidence:

- The French About and Privacy summaries are English (`content/fr/about.md:5`, `content/fr/privacy.md:10`).
- CV descriptions are inferred from mixed-language body text because the explicit descriptions are commented out.
- The global social image is a square 638 × 632 logo, while the templates request a large-image card; it may crop or display poorly on social platforms.
- PaperMod emits `BlogPosting` structured data for About, CV, and Privacy pages, although these are profile/document/policy pages.
- The configured regional language codes are nested under `params` (`config.yml:12` and `config.yml:46`), while rendered HTML uses the simpler `en` and `fr` language identifiers.

Impact: search snippets and link previews are less persuasive, French previews can appear in English, and structured data does not accurately describe the content.

Improvement:

- Add unique, localized `description` values to every page.
- Create a 1200 × 630 social card and test its crop on LinkedIn and common messaging clients.
- Override schema output so the homepage/About content uses `Person`/`ProfilePage` and policy/CV pages use an appropriate `WebPage` subtype.
- Place the regional `locale` at the language configuration level and consider an `x-default` hreflang for the default homepage.

#### 10. The portfolio content is too thin for a senior consultant

Evidence:

- The homepage communicates role and availability but not industries, technical strengths, outcomes, location/time zone, engagement type, or proof of work.
- The About pages are roughly one short screen and make dated claims (“currently” at Owkin, “since August 2023”, “starting ASAP”) with a 2025-02-05 content date.
- The navigation exposes only About; CV and contact are reachable mainly from homepage buttons.

Impact: a prospective client has to leave the site or open a PDF to understand fit, which weakens conversion and limits relevant search content.

Improvement:

- Confirm all current-employment and availability statements before editing them.
- Add concise sections for services, selected outcomes/case studies, stack, industries, engagement model, and contact expectations in both languages.
- Add clear CV and Contact navigation items and a stronger CTA such as “Discuss a data platform project”.
- Prefer measurable outcomes and constraints over a long technology list.

### P3 — Lower priority and engineering hygiene

#### 11. Security controls rely on HTML meta tags and allow inline code

Evidence:

- `layouts/partials/extend_head.html:7` sets CSP through a meta tag and permits `'unsafe-inline'` for scripts and styles.
- `layouts/partials/extend_head.html:15` sets Permissions Policy as a meta tag even though browser support for that delivery mechanism is limited; the template comment already acknowledges this.
- `frame-ancestors`, clickjacking protection, and several other controls require real HTTP response headers.

Impact: the CSP offers less protection than a nonce/hash-based policy, and some intended policies may not be enforced at all.

Improvement:

- Set CSP, Permissions Policy, `X-Content-Type-Options`, and clickjacking controls as Cloudflare/CDN response headers after inventorying required sources.
- Move inline scripts/styles into fingerprinted assets or adopt CSP hashes/nonces before removing `'unsafe-inline'`.
- Validate production headers after deployment rather than assuming meta tags are equivalent.

#### 12. CI deploys from `main` without pre-merge quality gates

Evidence:

- `.github/workflows/hugo.yml:4-10` runs on pushes to `main` and manual dispatch, not pull requests.
- The build does not fail on warnings and performs no link, HTML, accessibility, or visual checks.
- Several actions use mutable major-version tags, and some are still configured at `@v3`.
- The local Hugo version used in this audit was 0.155.2 while CI pins 0.149.0 (`.github/workflows/hugo.yml:37`).

Impact: the first comprehensive validation happens in the deployment workflow after a change reaches `main`, and local/CI behavior can drift.

Improvement:

- Add a pull-request check job that builds with `--panicOnWarning` and validates internal links.
- Add a small browser smoke test for English/French routing, page titles, `noindex`, missing assets, and mobile overflow.
- Add automated accessibility checks for the key routes.
- Align Hugo versions, keep actions current, and consider pinning third-party actions to reviewed commit SHAs.

## What is already working well

- The production-style Hugo build completed successfully with no Hugo i18n or path warnings.
- Canonical URLs and page-level English/French `hreflang` pairs are generated correctly.
- English and French pages render with correct `lang="en"` and `lang="fr"` attributes.
- Key pages had one clear H1 and no horizontal overflow at 1440px or 375px.
- The production homepage loaded its CSS and images without browser console errors during the spot check.
- The layout supports light/dark themes and respects reduced-motion preference for smooth scrolling.
- External LinkedIn links use `noopener`/`noreferrer`, and the generated CSS is fingerprinted.
- Both résumé PDFs are compact (about 52–53KB), despite needing accessibility metadata/tagging.

## Recommended implementation order

### First pass — highest return, low effort

- Remove unintended `noindex` directives.
- Fix translated route handling and localize shared UI strings.
- Rewrite the privacy text from the verified production configuration.
- Replace inline green styling with accessible theme-aware CSS.
- Add PDF titles/fallback links, accurate alt text, and meaningful link labels.
- Remove `.DS_Store`, unused taxonomies/search output, and the missing icon reference.

### Second pass — performance and metadata

- Move images into the Hugo asset pipeline and generate responsive formats.
- Add localized descriptions and a purpose-built social card.
- Improve structured data and regional language configuration.
- Regenerate accessible, tagged PDFs.

### Third pass — content and quality gates

- Expand the consultant portfolio content in English and French.
- Add PR build/link/accessibility smoke tests.
- Measure deployed response headers, Lighthouse lab metrics, and real Core Web Vitals.

## Suggested acceptance checks

- `/about/` and `/fr/about/` are indexable and mutually linked as translations.
- Switching language on About and CV preserves the current page.
- Every French route has French navigation, footer text, summary, date format, and calls to action.
- Dark-mode availability text reaches at least 4.5:1 contrast.
- Every non-decorative image and embedded document has an accurate accessible name.
- PDFs are tagged and still usable when inline PDF rendering is unavailable.
- No generated page references a missing local asset.
- No empty taxonomy/search artifacts or `.DS_Store` files are deployed.
- A pull request fails before merge when the Hugo build, internal link check, or accessibility smoke test fails.
